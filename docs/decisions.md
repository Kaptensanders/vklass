# Decisions

- 2026-03-26: `gateway.py` keeps unsupported auth modes (`login` and `file`) as explicit `NotImplementedError` stubs for now. This preserves fail-fast behavior while the standalone gateway work focuses on the cookie-callback path used by the sandbox test.
- 2026-03-26: `VklassSession` now keeps cookies as an in-memory dict for requests and merges cookies returned by successful Vklass responses into that dict. External cookie sources may still provide plaintext cookies, which are normalized on input.
- 2026-03-27: `VklassSession` keepalive now runs as a single background task started/stopped via `startKeepAlive()`/`stopKeepAlive()`. The keepalive ping uses authenticated fetch of `/Home/Welcome` with redirects disabled, reusing existing auth renewal/cookie update flow. Keepalive errors are logged and retried on next interval; task cancellation exits immediately.
- 2026-04-05: The active authentication design has changed from externally retrieving auth cookies to running the real Vklass authentication flow inside the standalone gateway session. BankID QR and BankID personal-number variants are now the intended primary design for guardian login, with username/password kept as a supported alternate path where districts allow it.
- 2026-04-05: BankID QR step 1 now derives later auth inputs from the live `logininitial` response in the shared `aiohttp` session. The initial response URL, base URL, query parameters, and HTML are stored in `authData` so later steps can avoid hardcoded district-specific values such as `Target` or `FedId`.
- 2026-04-05: `login.py` now loads auth adapters through the fully qualified package path (`custom_components.vklass.auth_adapters`) so adapter modules keep their package context and relative imports like `from ..const` work in both the integration and the sandbox bootstrap.
- 2026-04-05: The Göteborg BankID adapter now logs each QR/status poll iteration with HTTP metadata and truncated payloads, and it logs the full failure payload before returning `False` on BankID `failed` or `error` states. This keeps the flow fail-fast while making district-specific status failures visible.
- 2026-04-05: Authentication support is now defined by auth-adapter compatibility rather than a fixed list of supported methods. The configured auth URL is the primary selector, `login.py` autodetects a matching adapter via `can_handle(url)`, and a flow is supported only when a compatible module exists under `custom_components/vklass/auth_adapters/`.
- 2026-04-06: The Home Assistant config flow now always requires `auth_url`, treats all other auth fields as optional inputs, and checks adapter compatibility before creating the entry. When no adapter matches, the flow offers an explicit manual-cookie confirmation path that keeps `vklass.set_auth_cookie` as the supported fallback for now.
- 2026-04-06: `vklass.authenticate` now targets the Vklass auth sensor entity in Home Assistant. The user selects the `sensor.vklass_*_auth` entity by `entity_id`, while the integration resolves that entity back to the owning config entry and shared `VklassGateway`.
- 2026-04-06: Reconfiguring a Vklass entry now promotes the saved options values into the config entry `data` and clears `options`. This avoids restart-time drift where Home Assistant shows updated values in the options UI but the gateway still boots from stale setup-time auth settings.
- 2026-04-06: The authentication sensor now owns the gateway's auth-event and QR-code handlers. Integration setup keeps only the auth-cookie persistence handler in `__init__.py`, and the sensor platform is loaded before stored cookie restore so startup auth state is visible to the entity.
- 2026-04-06: Auth adapters now expose required metadata through `AUTH_METHOD` and `AUTH_INTERACTIVE`, and the auth sensor surfaces that as `auth_method` and `auth_interactive` attributes. When no adapter matches the configured URL, gateway-driven authentication still fails fast, while the Home Assistant UX may present the manual-cookie fallback.
- 2026-04-06: The Lovelace auth card now uses the auth sensor entity as its single target for both `vklass.authenticate` and `vklass.set_auth_cookie`. QR payloads remain raw in the gateway/sensor contract, and the integration renders them as an authenticated SVG image via `/api/vklass/qr`.
- 2026-04-06: The Göteborg BankID adapter now includes verbose handoff diagnostics around the post-BankID Göteborg and Vklass SAML chain, including cookie snapshots, `Set-Cookie` headers, redirect targets, and truncated HTML/body snippets. This is temporary observability work to reduce Home Assistant restarts while we align the gateway session with the browser flow.
- 2026-04-06: Göteborg IdP priming is now attempted after the successful BankID finish-form POST rather than before it. If that post-step5a prime already lands on the Göteborg IdP handoff page with the Vklass assertion form, the adapter reuses that page directly and skips the `/sp/sps/wssoi` follow-up.
- 2026-04-06: The Göteborg guardian BankID flow now bootstraps from `https://auth.vklass.se/organisation/189` before the QR sequence. The adapter reproduces the early browser chain through `auth.vklass.se/saml/initiate`, `authpub.goteborg.se/idp/sps/idppub/saml20/login`, and the pre-auth MGA redirect so the Göteborg IdP state and cookies are established before the existing `logininitial` and BankID steps.
- 2026-04-06: The attempted pre-QR bootstrap from `https://auth.vklass.se/organisation/189` was too brittle with a clean session and could fail before QR. The adapter now keeps the legacy `logininitial` path for QR extraction, hardcodes `organisation/189` only as a post-step5a replay, and uses the replayed `idppub` redirect target when available because the browser HAR shows that hop sets the missing Göteborg IdP cookie and includes the `PartnerId`/`FedName=idppub` handoff parameters.
- 2026-04-06: The post-step5a guardian replay now forces `IV_JCT=/idp` immediately before the replayed `idppub` login request. This is a narrow alignment with the successful browser HAR, which sends the `idppub` login request in IdP context rather than the SP context that the adapter still had after the BankID finish handoff.
- 2026-04-06: The guardian replay is now moved earlier in the session, immediately after the initial `logininitial` page fetch and before the QR flow continues. After that warm-up attempt, the adapter explicitly restores the Göteborg SP cookie context so the existing QR steps keep using the legacy `logininitial` path unchanged, while the later step-5 handoff only reuses any `idppub` auth target discovered by the early warm-up.
- 2026-04-06: The clean end-to-end HAR showed that Göteborg's missing `idppub` cookie is created before `logininitial`, from a cookie-clean `idppub` request reached via `custodian.vklass.se -> auth.vklass.se -> organisation/189?returnUrl=http://custodian.vklass.se/ -> saml/initiate`. The adapter now uses that Vklass-to-Göteborg bootstrap as the real pre-QR entry path, then keeps the existing QR and BankID steps once `logininitial_url` has been established.
- 2026-04-06: The clean bootstrap includes signed SAML redirect URLs from Vklass into Göteborg. Those redirects must be replayed without URL requoting or normalization, so the adapter now sends the raw `Location` value back to `aiohttp` as an encoded URL for the pre-QR Göteborg bootstrap hops.
- 2026-04-06: The successful clean bootstrap now also preserves the full Göteborg `idppub` auth redirect URL from the pre-QR handoff. Step 5 reuses that discovered URL after BankID completion so the later IdP prime keeps the HAR's `PartnerId` and `FedName=idppub` parameters instead of falling back to the plain `FedId` target.
- 2026-04-06: Once the clean bootstrap is in place and the `idppub` handoff state exists, the extra post-step5a IdP prime is no longer needed and can disturb the session before `/sp/sps/wssoi`. The adapter now skips that prime on the clean-bootstrap path and follows the HAR directly from the finish-form POST to `/sp/sps/wssoi`.
- 2026-04-06: On the clean-bootstrap path, step 5 now also skips manual reseeding of the Göteborg SP cookies (`IV_JCT` and `gbgTargetURL`) before posting the BankID finish form. The browser-shaped session already carries those values, and rewriting them late in the flow can perturb the `/sp/sps/wssoi` handoff.
- 2026-04-06: Skipping the late Göteborg SP cookie reseed before the BankID finish-form POST caused the finish POST itself to fail with `ServerDisconnectedError`. The adapter keeps the clean bootstrap and later handoff fixes, but restores the pre-step5a SP cookie reseed so step 5a remains stable while we continue narrowing the remaining `/sp/sps/wssoi` mismatch.
- 2026-04-06: The remaining blocker is now isolated to the post-`/sp/sps/wssoi` branch. Instead of hard-failing on a redirect from `wssoi` to Göteborg MGA, the adapter now follows that bounded post-`wssoi` redirect chain with raw encoded URLs, so we can either land on the IdP handoff page or see the exact branch dead-end in one run.
- 2026-04-06: The clean browser step-5 requests use Firefox-style navigation headers and explicit cross-site document fetch metadata on the finish POST, `wssoi`, and the final IdP handoff GET. The adapter now mirrors that header shape only for those late Göteborg requests and logs the exact outgoing header block, including the serialized cookie header, so the remaining `wssoi` mismatch can be compared byte-for-byte against the browser flow.
- 2026-04-06: The clean browser `Cookie` header order for the late Göteborg handoff is now mirrored explicitly in the adapter for step 5 and the post-`wssoi` follow-up. This keeps the request shape as close as possible to the HAR and lets us rule out brittle server behavior tied to cookie ordering.
- 2026-04-06: After the Göteborg handoff investigation, the adapter was cleaned up to remove the temporary replay/prime/MGA-follow experiments and the verbose success-path request-cookie tracing that did not lead to the auth cookie. The clean pre-QR bootstrap, the working QR flow, and the direct fail-fast `wssoi -> idp/sps/auth` expectation remain.
- 2026-04-06: The adapter now logs the actual prepared `aiohttp` request headers for the late Göteborg handoff requests (`step 5a`, `5b`, `5c`) using `resp.request_info.headers`. This is a narrow inspection aid so we can compare what `aiohttp` truly sent against the browser HAR before making further behavioral changes.
- 2026-04-06: The narrow wire-shape probe is now focused on the real choke point at `GET /sp/sps/wssoi` (`step 5b`). The adapter logs the prepared `aiohttp` request headers and the raw response headers for that request so we can compare the failing handoff directly against the browser flow.
- 2026-04-06: The late Göteborg handoff requests now use an explicit browser-shaped header block and a manually serialized `Cookie` header. This avoids `aiohttp` quoting values like `IV_JCT`, `gbgTargetURL`, and `PUBLIC-SESSION-ID`, and keeps the finish POST, `wssoi`, and final IdP GET closer to the browser HAR.
- 2026-04-06: The exact DevTools `Copy as cURL` shape for `GET /sp/sps/wssoi` is now exercised through a standalone sandbox script instead of the Home Assistant adapter. That keeps the adapter narrower while still giving us a reproducible way to test the wire shape independently.
- 2026-04-06: Added a standalone `wssoi` curl repro under `test/sandbox/` so the late Göteborg handoff can be tested independently of Home Assistant. The script keeps the DevTools curl shape fixed, and the env file holds only the dynamic cookie/target values to refresh from each new HA run.
- 2026-04-06: A second sandbox env now captures the latest `aiohttp`-serialized `wssoi` request shape from Home Assistant, including quoted cookie values. Running the standalone curl repro with the browser-shaped env succeeds, while running it with the `aiohttp`-shaped env reproduces the MGA redirect failure. This isolates the remaining blocker to request wire shape, especially cookie serialization, rather than the session values themselves.
- 2026-04-06: `GET /sp/sps/wssoi` now uses a one-off raw request path with `aiohttp.DummyCookieJar()` and an explicitly rendered curl-shaped `Cookie` header. Response cookies from that hop are merged back into the shared session jar so the rest of the Göteborg and Vklass SAML chain can continue unchanged if the raw `wssoi` handoff succeeds.
- 2026-04-06: `VklassSession` now owns its own `aiohttp.ClientSession` instead of receiving Home Assistant's shared session. The gateway session is configured with `CookieJar(quote_cookie=False)` and `requote_redirect_url=False` because the Göteborg BankID flow is proven to fail with default quoted-cookie serialization and to require raw signed redirect handling.
- 2026-04-06: After moving to a gateway-owned `aiohttp` session with `quote_cookie=False`, the temporary raw `wssoi` request workaround was removed again. The Göteborg adapter is back on normal `aiohttp` request handling so we can verify whether the session-level cookie-jar setting is sufficient on its own.
- 2026-04-06: The Göteborg adapter request handling was reviewed and simplified around the new gateway-owned session. The manual step-5 cookie header shaping and other proof-era request overrides were removed, and each request now sets only the per-call headers with real impact, mainly `Origin`, `Referer`, and `Content-Type`, while relying on the session defaults and normal `aiohttp` cookie handling for the rest.
- 2026-04-06: A second cleanup pass trimmed smaller Göteborg adapter leftovers from the investigation, including unused helper state, one-off header variables, and stale inline comments. The flow structure stays the same, but the request code now leans more on direct calls and fewer temporary indirections.
- 2026-04-06: The Göteborg adapter step helpers are now named in a single monotonic sequence (`step0` through `step10`) that matches the real auth flow order. Historical names like `step_minus1` and reused step numbers were removed to make the file easier to read and reason about.
- 2026-04-06: After the happy-path login was confirmed, the temporary Göteborg investigation logging was removed again. The adapter no longer logs bootstrap progress, step-5 request/response headers, or assertion-form payload summaries on success; it keeps only the logs that are still useful for normal flow visibility and failure diagnosis.
- 2026-04-06: The Göteborg IdP handoff page is now parsed in the same step that fetches it. The separate assertion-form extraction step was removed so raw HTML is not carried across steps, and the final Vklass assertion post is now `step9`.
- 2026-04-07: `VklassGateway.logout()` is now exposed as the Home Assistant service `vklass.logout`, targeting the same Vklass auth sensor entity pattern as `vklass.authenticate` and `vklass.set_auth_cookie`.
- 2026-04-09: Test bootstrap under `test/tests/helpers/bootstrap.py` is now limited to adding the repo and `test/` directories to `sys.path`. The old dynamic `vklassgateway` alias and missing-dependency stubs were removed so tests use the real package import path `custom_components.vklass.vklassgateway`, matching editor and runtime behavior.
- 2026-04-09: Added `test/__init__.py` so repo-local imports like `from test.tests.helpers import bootstrap` resolve as a real package path. This avoids collisions with Home Assistant Core's separate `tests` package while keeping the local test helper layout explicit.
- 2026-04-11: Authentication design is now adapter-selected rather than auth-URL-driven. The public gateway API is `VklassGateway.login(credentials=None, reuse_credentials=False)`, while `_authenticate(...)` remains internal. Home Assistant persists reusable auth data outside the gateway, reseeds it through `login(..., reuse_credentials=True)`, and exposes only safe auth metadata on the auth sensor. Persisted password and cookie fields use the sentinel `__PERSISTED_SECRET__` in the card, and the backend swaps that placeholder back to the stored secret before calling the gateway.
- 2026-04-11: The Home Assistant integration now persists only adapter-specific runtime credentials selected by the user through `save_credentials`. The auth sensor exposes safe rendering metadata (`auth_method`, `auth_adapter`, visible identifiers, and persisted-secret booleans), while the card submits credentials through `vklass.authenticate` and reuses stored password/cookie values only when the sentinel `__PERSISTED_SECRET__` is posted back unchanged.
- 2026-04-12: The auth sensor now exposes `device_name` and `auth_adapter_title` as safe UI metadata. The Lovelace auth card always renders `device_name` as its header, renders `auth_adapter_title` as a subtitle only on logged-out login screens, and now uses the standard Lovelace error card for configuration/runtime card errors such as missing entities or unsupported auth methods.
- 2026-04-12: The Home Assistant config flow no longer exposes `keepalive_minutes`. Keepalive is now treated as a static integration setting from `DEFAULT_KEEPALIVE_MINUTES`, and `async_setup_entry` always seeds that default into the config passed to `VklassGateway` when no explicit value exists.
- 2026-04-12: Home Assistant startup restore now prefers `resumeLoggedInSession()` when a persisted auth cookie exists. If cookie-based resume returns `False`, startup falls back to `login(..., reuse_credentials=True)` only when reusable credentials are still available for that entry.
- 2026-04-12: Persisted Vklass auth cookies are now treated as session state rather than saved credentials. Refreshed cookies are always persisted and always tried through `resumeLoggedInSession()` at startup regardless of auth method or `save_credentials`, while credential-based startup `login(..., reuse_credentials=True)` still depends on saved reusable credentials. Manual-cookie auth no longer uses the `save credentials` UI path, and any auth `FAIL` clears the persisted cookie for that device.
- 2026-04-12: The Home Assistant auth-state policy was extracted into `custom_components/vklass/auth_state.py`. `__init__.py` now focuses on integration wiring and handler registration, while the auth card keeps its UX but centralizes auth-method-specific field and payload rules in small helper methods instead of scattering those branches across rendering and submit paths.
- 2026-04-12: The Home Assistant service previously exposed as `vklass.authenticate` is now `vklass.login`. The service behavior and payload shape stay the same; only the public service name changed to match `VklassGateway.login(...)`.
- 2026-04-12: Explicit logout is now a full sign-out policy in Home Assistant, not only a runtime session drop. Logout must clear persisted credentials, persisted auth cookie, and persistence preference for the entry so no automatic resume or auto-login happens until the user explicitly logs in again. Runtime auth state in Home Assistant is the authority for gating entity updates, while entities stay auth-agnostic and ask a runtime policy helper whether fetching is currently allowed. The auth sensor remains a UI/debug view of that state, not the coordination mechanism for other entities.
- 2026-04-13: The auth sensor now always exposes a `user` attribute. It is an empty string when no logged-in Vklass session context is available, and the Lovelace auth card renders that logged-in display name beneath the adapter title after successful authentication.
- 2026-04-13: Codex devcontainer state is persisted through a repo-local bind mount from `${localWorkspaceFolder}/.codex_cache` to `/home/vscode/.codex`. The repository keeps `.codex_cache/readme.md` tracked so the mount source exists after checkout, while `.gitignore` ignores the live Codex auth/session files created there.
- 2026-04-13: VS Code configuration ownership is split by scope. `workspace.code-workspace` keeps only multi-root workspace concerns (`folders` and excludes for the extra `ha_core`/`ha_config` roots), `.vscode/settings.json` keeps repo-level editor behavior and repo-root excludes, and `.devcontainer/devcontainer.json` keeps container setup plus container-specific editor settings such as the interpreter path and analysis extra paths.
- 2026-04-13: The Vklass auth card frontend is now served from `/vklass/vklass-auth-card.js` and treated as a normal Lovelace `module` resource in storage-mode resource setups. Integration setup matches resources by the base path without query parameters. If multiple Vklass card resources already exist, the integration removes all matching entries and recreates one canonical resource for the current version; if exactly one old entry exists, it is updated in place. YAML-mode Lovelace resources are not mutated; those users must declare the resource explicitly in their YAML config.
- 2026-04-13: `VklassGateway.getCalendar()` now translates `dateBegin` and `dateEnd` into Sweden-local ISO datetime strings before the calendar request, for example `2026-04-13T00:00:00+02:00`. The gateway accepts either `date` objects or ISO `YYYY-MM-DD` strings for those inputs and fails fast on other types.
- 2026-04-13: Calendar entities will be discovered dynamically from fetched Vklass events using bucket keys based on `(context, eventType)`. Friendly names are `<context> - <event type label>` using `CALENDAR_EVENTTYPES`, except any `context = null` events belong to the shared bucket `Vklass Helgdagar` regardless of `eventType`. Raw Vklass calendar payload must be normalized before it reaches the Home Assistant calendar layer, and normalized all-day events must follow Home Assistant's exclusive-end-date semantics.
- 2026-04-13: Normalized calendar event descriptions will use plain text only and preserve semantic line breaks as `\n`. Raw Vklass HTML formatting is not preserved in normalized events, but HTML line-break and block structure may be converted into plain-text newlines before tags are removed so both the standard Home Assistant calendar view and richer custom calendar cards can consume the same `description` field.
- 2026-04-13: Grouped calendar bucket metadata now keeps the raw `event_type` for completeness and debugging, including the shared `Vklass Helgdagar` bucket. That field is informational only for the shared null-context bucket and is not part of that bucket's identity. Calendar normalization is tolerant of malformed individual source entries: bad events are skipped with warnings instead of failing the entire calendar fetch.
- 2026-04-19: The `vklass_userpass` adapter now mirrors the documented working username/password flow from `docs/vklass_sensor_example.py` inside the shared gateway `aiohttp` session. It fetches the credentials page to extract `__RequestVerificationToken`, posts to `https://auth.vklass.se/credentials/signin`, treats `302` as a successful handoff, and leaves the final authenticated-session validation to `VklassSession._verifyAuth()`.
- 2026-04-19: Calendar refresh warnings now log the Vklass device name and related calendar targets instead of only the opaque config-entry id. The runtime prefers real registered `calendar.*` entity ids when available, and falls back to discovered bucket names before entity registration has completed.
- 2026-04-19: Startup ordering now restores Vklass runtime auth before the calendar platform is set up. The auth sensor is still loaded first so startup auth state is visible, but calendar entities are not created until after `resumeLoggedInSession()` or credential-based restore has been attempted. Entity fetch gating now requires a live loaded gateway context rather than treating a persisted cookie in storage as fetch-ready on its own.
- 2026-04-19: Calendar fetch logging now comes from the Home Assistant calendar runtime instead of the gateway's month-range log line. The log uses the Vklass device name plus the requested month as `YYYY-MM`, for example `<device name> requesting calendar data for 2026-04`.
- 2026-04-19: The Vklass auth card now suppresses keyboard shortcut propagation from all form inputs inside the card, and it explicitly cancels search-like shortcuts such as `/` and `Ctrl/Cmd+K`. This prevents Home Assistant global search from stealing focus while the user types credentials.
- 2026-04-19: The auth card no longer re-renders its full shadow DOM on every form keystroke. Credential inputs now update only local form state plus the authenticate button's disabled state, so focus stays in the same input element and Home Assistant global hotkeys do not receive the next keystroke after a typed character.
- 2026-04-19: The stale keepalive tests were updated to the current `VklassSession` API (`_startKeepAlive`, `_stopKeepAlive`, `_keepAliveLoop`, adapter-based config), and placeholder test modules were made import-safe so pytest collection no longer depends on dead legacy imports. The design doc was also aligned with the current public Home Assistant service name `vklass.login` and with the shipped manual-cookie UX, which no longer exposes saved-cookie sentinel handling or a `save credentials` checkbox.
- 2026-04-19: Calendar refreshes triggered by runtime auth-state callbacks are now edge-triggered on `can_entry_fetch()` becoming allowed instead of firing on every runtime-state write. This avoids the duplicate 12-month refresh immediately after a successful interactive login, where the integration first broadcasts auth success and then broadcasts persisted post-login state such as `save_credentials`.
- 2026-04-19: Hassfest now declares the integration's Home Assistant component relationships explicitly: `http` is a hard dependency because setup registers an authenticated QR view and static frontend path, while `lovelace` is an `after_dependency` because the integration only opportunistically syncs a Lovelace module resource when that subsystem is available. The integration also declares `CONFIG_SCHEMA = cv.config_entry_only_config_schema(DOMAIN)` because it uses `async_setup` for service/frontend wiring but has no YAML configuration.
